* Generic IMAP SMTP Baseline
This plan should produce the first usable provider slice for generic email accounts. The outcome is not "protocol client code exists"; it is a storage-backed baseline that can connect to a generic IMAP account, ingest a bounded mailbox slice into local storage, and send a basic SMTP message without redefining account or storage foundations.

** Goals For This Plan
- [ ] Implement a generic IMAP adapter in ~meeru-providers~ that can connect, authenticate, list folders, and fetch messages for one selected mailbox
- [ ] Parse fetched RFC822 message bytes with ~mail-parser~ and convert them into the storage-ready email shape used by ~meeru-core~
- [ ] Persist first-pass synced messages, bodies, folder membership, and attachment payloads through the existing storage interfaces from plan 002
- [ ] Implement a basic SMTP send path for generic accounts
- [ ] Keep the slice intentionally narrow enough that plan 005 can build account orchestration and multi-account sync on top without reworking provider primitives

** Scope For V1
*** Included In V1
- [ ] Generic IMAP session establishment using account configuration supplied by the shared account/auth layer
- [ ] IMAP operations for:
  - [ ] connect and authenticate
  - [ ] list folders
  - [ ] fetch a bounded page of message summaries
  - [ ] fetch full RFC822 message content for selected messages
  - [ ] basic server-side search where the protocol surface supports it
- [ ] RFC822 parsing with ~mail-parser~ for:
  - [ ] headers needed by the current email model
  - [ ] plain text body extraction
  - [ ] HTML body extraction when present
  - [ ] attachment metadata and payload extraction
- [ ] Conversion from parsed provider data into the core/storage sync shape
- [ ] SMTP send of a simple composed message using the same account configuration
- [ ] Integration tests using representative MIME fixtures and storage round-trip behavior

*** Deferred From V1
- Provider-specific Gmail or Outlook behavior
- Multi-account scheduling and concurrent sync orchestration
- Push or near-real-time update support such as IMAP IDLE
- Delta sync features such as CONDSTORE or provider-specific sync tokens
- Connection pooling beyond a resilient per-account session lifecycle
- Full draft sync, sent-folder reconciliation, and conflict resolution workflows

** Implementation Plan
*** 1. Provider Runtime Boundary
- [ ] Define the provider-facing configuration shape required for generic IMAP and SMTP access
- [ ] Make the generic adapter consume shared account/auth data rather than inventing a parallel configuration path
- [ ] Keep the low-level protocol code contained within ~meeru-providers~

Acceptance criteria:
- [ ] The generic adapter can be constructed from the shared account/runtime configuration without SQLite-specific types
- [ ] Plan 005 can reuse the adapter without changing its public contract

*** 2. IMAP Fetch Slice
- [ ] Implement connection establishment, authentication, and folder listing
- [ ] Implement bounded message listing for one selected provider folder
- [ ] Fetch raw RFC822 bytes for selected messages

Acceptance criteria:
- [ ] A generic account can list at least one mailbox and fetch message content from it
- [ ] IMAP failures return typed errors with actionable protocol context

*** 3. Message Parsing And Conversion
- [ ] Parse headers, addresses, dates, text bodies, HTML bodies, and attachments from RFC822 bytes
- [ ] Convert parsed messages into the storage-ready sync shape used by ~meeru-core~
- [ ] Preserve provider-native identifiers required for later re-sync

Acceptance criteria:
- [ ] Parsed messages populate the existing email and attachment storage models without ad-hoc field loss
- [ ] MIME fixture tests cover multipart, HTML-only, text-only, and attachment-bearing messages

*** 4. Storage-Backed First Sync
- [ ] Use existing storage services to cache fetched email bodies and attachments
- [ ] Map fetched provider folders into unified folder assignments using the current folder mapping primitives
- [ ] Define duplicate-handling behavior for already-seen provider message IDs

Acceptance criteria:
- [ ] First-pass sync can ingest a bounded mailbox slice into local storage
- [ ] Stored bodies and attachments can be read back through existing storage APIs

*** 5. SMTP Send Baseline
- [ ] Implement an SMTP client wrapper for generic accounts
- [ ] Send a basic outbound message
- [ ] Define the V1 local follow-up behavior after a successful send

Acceptance criteria:
- [ ] A simple outbound message can be submitted through SMTP with typed error handling
- [ ] The plan documents whether sent-mail persistence is immediate, deferred, or handled by later sync

*** 6. Test Coverage
- [ ] Replace placeholder provider tests with protocol and parsing focused coverage
- [ ] Add end-to-end tests that exercise parsing plus storage persistence for representative messages

Acceptance criteria:
- [ ] ~cargo test~ exercises real message parsing and storage-backed sync behavior
- [ ] The generic provider slice is usable without provider-specific code paths

** Exit Criteria
- [ ] ~meeru-providers~ exposes a working generic IMAP and SMTP baseline rather than empty modules
- [ ] Generic account sync can populate local storage using the plan 002 interfaces
- [ ] Generic email sending works for the supported authentication/configuration shape
- [ ] Plan 005 can focus on account orchestration instead of redesigning provider primitives
