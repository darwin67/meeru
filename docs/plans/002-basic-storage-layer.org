* Basic Storage Layer
This plan should produce the first executable storage slice for Meeru. The outcome is not "schema written on paper"; it is a working local persistence layer that later IMAP sync, UI, and CLI work can use without redesigning storage a second time.

** Goals For This Plan
- [ ] Bootstrap a local storage runtime that can open an app data directory, create a SQLite database, and run migrations automatically
- [ ] Persist account metadata, unified folder metadata, email metadata, and attachment metadata in SQLite
- [ ] Persist email bodies and attachment payloads in the local file system with stable references from SQLite
- [ ] Expose storage traits and a concrete SQLite implementation that ~meeru-core~, ~meeru-cli~, and future provider adapters can call
- [ ] Cover the storage layer with integration tests that exercise a real database and real on-disk content storage

** Scope For V1
*** Included In V1
- [ ] SQLite schema and migration runner for:
  - [ ] ~schema_migrations~
  - [ ] ~accounts~
  - [ ] ~unified_folders~
  - [ ] ~folder_mappings~
  - [ ] ~emails~
  - [ ] ~email_folders~
  - [ ] ~attachments~
- [ ] Database bootstrap and connection pool management
- [ ] Storage traits for account, folder, email, and attachment operations
- [ ] SQLite-backed repositories for the V1 entities
- [ ] File-system blob storage for email content and attachment bytes
- [ ] Integration tests for migration, CRUD, query, and blob round-trip behavior

*** Deferred From V1
- [ ] Encryption at rest for stored email content
- [ ] Backup and restore workflows
- [ ] Tantivy search index integration
- [ ] Contacts, labels, AI task tables, audit log, search history, and other non-core metadata tables
- [ ] Conflict resolution and sync queue persistence

** Implementation Plan
*** 1. Storage Bootstrap
- [ ] Define storage path conventions for database, blob content, temporary files, and future backups
- [ ] Implement a storage configuration type that can open a workspace-local or user-data-local storage root
- [ ] Add a database bootstrap API in ~meeru-storage~ that:
  - [ ] creates required directories
  - [ ] opens the SQLite database
  - [ ] configures SQLite pragmas appropriate for a desktop app
  - [ ] creates a reusable connection pool

Acceptance criteria:
- [ ] Starting from an empty directory creates the required storage folders and database file
- [ ] Opening the same directory twice reuses the existing schema without data loss
- [ ] Bootstrap failures return typed errors with actionable context

*** 2. Migration System
- [ ] Implement a migration runner in ~meeru-storage::migrations~
- [ ] Create an initial executable V1 migration derived from ~docs/database-schema.org~
- [ ] Keep migrations idempotent and versioned through ~schema_migrations~
- [ ] Validate the documented schema against actual SQLite syntax before checking off the migration work

Acceptance criteria:
- [ ] A fresh database applies all migrations successfully
- [ ] Re-running migrations on an up-to-date database is a no-op
- [ ] The migration runner records applied versions and descriptions
- [ ] Migration tests fail if the documented schema and executable migration drift in incompatible ways

*** 3. Storage Model Alignment
- [ ] Define storage-layer record types or mapping functions for the current core models in ~meeru-core~
- [ ] Resolve mismatches between the current lightweight domain structs and the richer storage schema
- [ ] Document which fields are required in V1, optional in V1, or deferred to later plans

Acceptance criteria:
- [ ] Account and email records can be converted between storage and core types without lossy ad-hoc parsing
- [ ] Deferred columns are explicit rather than silently unused

*** 4. Storage Traits
- [ ] Define focused traits or repository interfaces for:
  - [ ] account CRUD
  - [ ] unified folder CRUD and folder mapping lookup
  - [ ] email insert/update/get/list operations
  - [ ] attachment insert/list operations
  - [ ] content blob put/get/delete operations
- [ ] Keep trait boundaries small enough that future provider sync code can compose them cleanly

Acceptance criteria:
- [ ] ~meeru-core~ can depend on storage interfaces without importing SQLite-specific types
- [ ] Trait methods map to realistic workflows needed by plan 003 and plan 004

*** 5. SQLite Backend
- [ ] Implement the V1 repository traits using ~sqlx~ and SQLite
- [ ] Support basic transactional operations for writing an email record and its related folder and attachment metadata
- [ ] Add list/query methods needed for an initial inbox or folder view

Acceptance criteria:
- [ ] Can create, list, fetch, update, and delete accounts
- [ ] Can insert and fetch email metadata by account and folder
- [ ] Can associate emails to unified folders
- [ ] Can persist and fetch attachment metadata linked to an email

*** 6. File-System Blob Store
- [ ] Implement local blob storage for raw email bodies
- [ ] Implement local blob storage for attachment payloads
- [ ] Store stable relative references in SQLite rather than embedding large payloads in the database
- [ ] Define cleanup behavior for blob writes that fail partway through a transaction

Acceptance criteria:
- [ ] Email body content can be written and read back by reference
- [ ] Attachment payloads can be written and read back by reference
- [ ] Blob paths are deterministic or otherwise traceable for debugging

*** 7. Test Coverage
- [ ] Replace placeholder crate tests with storage-focused unit and integration tests
- [ ] Use temp directories and a real SQLite database in tests
- [ ] Cover migration bootstrap, account CRUD, email CRUD, folder mapping, attachment metadata, and blob round-trip behavior

Acceptance criteria:
- [ ] ~cargo test~ exercises real storage behavior rather than placeholder assertions
- [ ] Tests can run repeatedly without manual cleanup

** Exit Criteria
- [ ] ~meeru-storage~ exposes a documented bootstrap API and concrete storage implementation
- [ ] The workspace has a real migration-backed SQLite database instead of empty module files
- [ ] The storage layer supports the minimum data flows needed for generic account setup and first-pass email sync work
- [ ] Plan 003 can build on the storage interfaces without redefining schemas or persistence primitives

** Notes
- [ ] Keep this plan intentionally narrower than the full schema document. The schema doc is a design reference, not a requirement to implement every table before the first usable storage layer exists.
- [ ] Do not mark schema design items done solely because SQL exists in documentation; the executable migration is the source of truth for implementation status.
- [ ] Treat encryption, backup, search indexing, and broader metadata tables as follow-on work unless one becomes a hard dependency for the first sync or read flow.
