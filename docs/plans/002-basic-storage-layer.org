* Basic Storage Layer
This plan should produce the first executable storage slice for Meeru. The outcome is not "schema written on paper"; it is a working local persistence layer that later IMAP sync, UI, and CLI work can use without redesigning storage a second time.
The storage model for Meeru should support both folder-based and label-based providers. Folders are the primary navigation container; labels are separate many-to-many classification metadata. Provider-native folder IDs and label IDs should be preserved independently so sync adapters can round-trip Gmail-style labels and IMAP-style folders without collapsing one concept into the other.

** Goals For This Plan
- [X] Bootstrap a local storage runtime that can open an app data directory, create a SQLite database, and run migrations automatically
- [X] Persist account metadata, unified folder metadata, email metadata, and attachment metadata in SQLite
- [X] Persist email bodies and attachment payloads in the local file system with stable references from SQLite
- [X] Expose storage traits and a concrete SQLite implementation that ~meeru-core~, ~meeru-cli~, and future provider adapters can call
- [X] Cover the storage layer with integration tests that exercise a real database and real on-disk content storage
- [X] Keep the V1 storage slice compatible with a later first-class label model instead of baking label semantics into folder-only abstractions

** Scope For V1
*** Included In V1
- [X] SQLite schema and migration runner for:
  - [X] ~schema_migrations~
  - [X] ~accounts~
  - [X] ~unified_folders~
  - [X] ~folder_mappings~
  - [X] ~emails~
  - [X] ~email_folders~
  - [X] ~attachments~
- [X] Database bootstrap and connection pool management
- [X] Storage traits for account, folder, email, and attachment operations
- [X] SQLite-backed repositories for the V1 entities
- [X] File-system blob storage for email content and attachment bytes
- [X] Integration tests for migration, CRUD, query, and blob round-trip behavior
- [X] A storage shape that treats folders as navigation containers and leaves room for later ~labels~ and ~email_labels~ tables without reworking the V1 email or account foundations

*** Deferred From V1
- Encryption at rest for stored email content
- Backup and restore workflows
- Tantivy search index integration
- Full first-class label persistence (~labels~, ~email_labels~, and provider-label mapping tables)
- Contacts, AI task tables, audit log, search history, and other non-core metadata tables
- Conflict resolution and sync queue persistence

** Implementation Plan
*** 1. Storage Bootstrap
- [X] Define storage path conventions for database, blob content, temporary files, and future backups
- [X] Implement a storage configuration type that can open a workspace-local or user-data-local storage root
- [X] Add a database bootstrap API in ~meeru-storage~ that:
  - [X] creates required directories
  - [X] opens the SQLite database
  - [X] configures SQLite pragmas appropriate for a desktop app
  - [X] creates a reusable connection pool

Acceptance criteria:
- [X] Starting from an empty directory creates the required storage folders and database file
- [X] Opening the same directory twice reuses the existing schema without data loss
- [X] Bootstrap failures return typed errors with actionable context

*** 2. Migration System
- [X] Implement a migration runner in ~meeru-storage::migrations~
- [X] Create an initial executable V1 migration derived from ~docs/database-schema.org~
- [X] Keep migrations idempotent and versioned through ~schema_migrations~
- [X] Validate the documented schema against actual SQLite syntax before checking off the migration work

Acceptance criteria:
- [X] A fresh database applies all migrations successfully
- [X] Re-running migrations on an up-to-date database is a no-op
- [X] The migration runner records applied versions and descriptions
- [X] Migration tests fail if the documented schema and executable migration drift in incompatible ways

*** 3. Storage Model Alignment
- [X] Define storage-layer record types or mapping functions for the current core models in ~meeru-core~
- [X] Resolve mismatches between the current lightweight domain structs and the richer storage schema
- [X] Document which fields are required in V1, optional in V1, or deferred to later plans
- [X] Make the V1 model boundaries explicit: folders are navigation containers, labels are a later independent concept, and provider-native mappings should stay separate for each

Acceptance criteria:
- [X] Account and email records can be converted between storage and core types without lossy ad-hoc parsing
- [X] Deferred columns are explicit rather than silently unused
- [X] The documented V1 shape can evolve to support both folder-first and label-first providers without redefining accounts, emails, or blob storage

*** 4. Storage Traits
- [X] Define focused traits or repository interfaces for:
  - [X] account CRUD
  - [X] unified folder CRUD and folder mapping lookup
  - [X] email insert/update/get/list operations
  - [X] attachment insert/list operations
  - [X] content blob put/get/delete operations
- [X] Keep trait boundaries small enough that future provider sync code can compose them cleanly

Acceptance criteria:
- [X] ~meeru-core~ can depend on storage interfaces without importing SQLite-specific types
- [X] Trait methods map to realistic workflows needed by plan 003 and plan 004
- [X] Folder-oriented V1 traits do not imply that labels must later be represented as folders

*** 5. SQLite Backend
- [X] Implement the V1 repository traits using ~sqlx~ and SQLite
- [X] Support basic transactional operations for writing an email record and its related folder and attachment metadata
- [X] Add list/query methods needed for an initial inbox or folder view

Acceptance criteria:
- [X] Can create, list, fetch, update, and delete accounts
- [X] Can insert and fetch email metadata by account and folder
- [X] Can associate emails to unified folders
- [X] Can persist and fetch attachment metadata linked to an email

*** 6. File-System Blob Store
- [X] Implement local blob storage for raw email bodies
- [X] Implement local blob storage for attachment payloads
- [X] Store stable relative references in SQLite rather than embedding large payloads in the database
- [X] Define cleanup behavior for blob writes that fail partway through a transaction

Acceptance criteria:
- [X] Email body content can be written and read back by reference
- [X] Attachment payloads can be written and read back by reference
- [X] Blob paths are deterministic or otherwise traceable for debugging

*** 7. Test Coverage
- [X] Replace placeholder crate tests with storage-focused unit and integration tests
- [X] Use temp directories and a real SQLite database in tests
- [X] Cover migration bootstrap, account CRUD, email CRUD, folder mapping, attachment metadata, and blob round-trip behavior

Acceptance criteria:
- [X] ~cargo test~ exercises real storage behavior rather than placeholder assertions
- [X] Tests can run repeatedly without manual cleanup

** Exit Criteria
- [X] ~meeru-storage~ exposes a documented bootstrap API and concrete storage implementation
- [X] The workspace has a real migration-backed SQLite database instead of empty module files
- [X] The storage layer supports the minimum data flows needed for generic account setup and first-pass email sync work
- [X] Plan 003 can build on the storage interfaces without redefining schemas or persistence primitives

** Notes
- Keep this plan intentionally narrower than the full schema document. The schema doc is a design reference, not a requirement to implement every table before the first usable storage layer exists.
- Do not mark schema design items done solely because SQL exists in documentation; the executable migration is the source of truth for implementation status.
- Treat encryption, backup, search indexing, and broader metadata tables as follow-on work unless one becomes a hard dependency for the first sync or read flow.
- Meeru's long-term organization model is dual: folders for navigation, labels for classification. V1 intentionally implements the folder slice first because it is enough to bootstrap local storage and initial mailbox flows, but later plans should add labels as a separate first-class concept rather than overloading ~unified_folders~ or ~folder_mappings~.
