* Overview
Meeru uses SQLite as its primary database for storing email metadata, account information, and application state. The actual email content is stored in encrypted files on disk with references in the database.

* Schema Version Management

    version_id INTEGER NOT NULL,
    is_applied INTEGER NOT NULL,
    tstamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

* Core Tables

** Accounts Table
Stores email account configurations and authentication details.

#+begin_src sql
CREATE TABLE accounts (
    id TEXT PRIMARY KEY,  -- UUID
    email TEXT NOT NULL UNIQUE,
    display_name TEXT,
    provider_type TEXT NOT NULL CHECK (provider_type IN ('gmail', 'outlook', 'generic')),

    -- Connection settings
    imap_host TEXT,
    imap_port INTEGER,
    imap_security TEXT CHECK (imap_security IN ('tls', 'starttls', 'none')),
    smtp_host TEXT,
    smtp_port INTEGER,
    smtp_security TEXT CHECK (smtp_security IN ('tls', 'starttls', 'none')),

    -- Authentication (encrypted)
    auth_method TEXT CHECK (auth_method IN ('password', 'oauth2')),
    encrypted_credentials TEXT,  -- Encrypted JSON with credentials
    oauth_refresh_token TEXT,    -- Encrypted
    oauth_access_token TEXT,     -- Encrypted
    oauth_token_expires TIMESTAMP,

    -- Sync settings
    sync_enabled BOOLEAN DEFAULT 1,
    sync_interval_minutes INTEGER DEFAULT 15,
    last_sync_time TIMESTAMP,
    last_sync_status TEXT,

    -- Metadata
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_active BOOLEAN DEFAULT 1,

    -- Provider-specific settings (JSON)
    provider_settings TEXT
);

CREATE INDEX idx_accounts_email ON accounts(email);
CREATE INDEX idx_accounts_active ON accounts(is_active);
#+end_src

** Folder Mappings Table
Maps unified folders to provider-specific folders.
This table is for folder-style provider containers only. Label-style provider metadata should be modeled separately instead of being forced through folder mappings.

#+begin_src sql
CREATE TABLE folder_mappings (

    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

** Emails Table
Stores email metadata with references to content files.

#+begin_src sql
CREATE TABLE emails (
    id TEXT PRIMARY KEY,  -- UUID
    account_id TEXT NOT NULL REFERENCES accounts(id),
    provider_id TEXT NOT NULL,  -- Provider's email ID (UID for IMAP)
    message_id TEXT,  -- RFC Message-ID header
    thread_id TEXT,   -- For conversation threading

    -- Email headers
    subject TEXT,
    from_address TEXT,
    from_name TEXT,
    to_addresses TEXT,    -- JSON array
    cc_addresses TEXT,    -- JSON array
    bcc_addresses TEXT,   -- JSON array
    reply_to TEXT,

    -- Dates
    date_sent TIMESTAMP,
    date_received TIMESTAMP,
    date_internal TIMESTAMP,  -- Internal timestamp for sorting

    -- Content references
    content_file_path TEXT,   -- Encrypted file path
    content_hash TEXT,        -- SHA-256 of content
    size_bytes INTEGER,
    has_attachments BOOLEAN DEFAULT 0,
    attachment_count INTEGER DEFAULT 0,

    -- Flags and status
    is_read BOOLEAN DEFAULT 0,
    is_starred BOOLEAN DEFAULT 0,
    is_important BOOLEAN DEFAULT 0,
    is_draft BOOLEAN DEFAULT 0,
    is_sent BOOLEAN DEFAULT 0,
    is_deleted BOOLEAN DEFAULT 0,

    -- AI-generated metadata
    ai_category TEXT,
    ai_priority INTEGER,
    ai_summary TEXT,
    ai_sentiment TEXT,
    ai_processed_at TIMESTAMP,

    -- Search and indexing
    search_text TEXT,  -- Preprocessed searchable content
    tantivy_doc_id TEXT,  -- Reference to full-text search index

    -- Sync metadata
    sync_state TEXT CHECK (sync_state IN ('synced', 'pending_local', 'pending_remote', 'conflict')),
    last_modified TIMESTAMP,

    -- Metadata
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    UNIQUE(account_id, provider_id)
);

CREATE INDEX idx_emails_account ON emails(account_id);
CREATE INDEX idx_emails_thread ON emails(thread_id);
CREATE INDEX idx_emails_message_id ON emails(message_id);
CREATE INDEX idx_emails_from ON emails(from_address);
CREATE INDEX idx_emails_date ON emails(date_internal DESC);
CREATE INDEX idx_emails_unread ON emails(is_read, account_id);
CREATE INDEX idx_emails_starred ON emails(is_starred, account_id);
CREATE INDEX idx_emails_search ON emails(subject, from_address, from_name);
#+end_src

** Labels Table
Unified label system used for cross-provider classification metadata.
Labels are distinct from unified folders: folders drive navigation, while labels are many-to-many metadata that can coexist with folders.

#+begin_src sql
CREATE TABLE labels (
    id TEXT PRIMARY KEY,  -- UUID
    name TEXT NOT NULL,
    color TEXT,
    icon TEXT,

    -- Hierarchy
    parent_id TEXT REFERENCES labels(id),
    path TEXT,  -- Full path like "Work/Projects/ClientA"

    -- Metadata
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    UNIQUE(path)
);

CREATE INDEX idx_labels_parent ON labels(parent_id);
CREATE INDEX idx_labels_path ON labels(path);
#+end_src

* Views

** Email List View
Optimized view for displaying email lists.

#+begin_src sql
CREATE VIEW email_list_view AS
SELECT
    e.id,
    e.account_id,
    a.email as account_email,
    e.subject,
    e.from_name,
    e.from_address,
    e.date_internal,
    e.is_read,
    e.is_starred,
    e.is_important,
    e.has_attachments,
    e.ai_category,
    e.ai_priority,
    GROUP_CONCAT(DISTINCT l.name) as labels,
    GROUP_CONCAT(DISTINCT uf.name) as folders
FROM emails e
JOIN accounts a ON e.account_id = a.id
LEFT JOIN email_labels el ON e.id = el.email_id
LEFT JOIN labels l ON el.label_id = l.id
LEFT JOIN email_folders ef ON e.id = ef.email_id
LEFT JOIN unified_folders uf ON ef.unified_folder_id = uf.id
WHERE e.is_deleted = 0
GROUP BY e.id;
#+end_src

** Contact Statistics View
Aggregated contact information.

#+begin_src sql
CREATE VIEW contact_stats_view AS
SELECT
    c.id,
    c.display_name,
    c.primary_email,
    c.is_vip,
    c.email_count_sent + c.email_count_received as total_emails,
    CASE
        WHEN c.last_email_received > c.last_email_sent OR c.last_email_sent IS NULL
        THEN c.last_email_received
        ELSE c.last_email_sent
    END as last_contact,
    COUNT(DISTINCT e.id) as actual_email_count
FROM contacts c
LEFT JOIN emails e ON (e.from_address = c.primary_email OR e.to_addresses LIKE '%' || c.primary_email || '%')
GROUP BY c.id;
#+end_src

* Indexes Strategy

** Design Principles
1. *Primary Keys*: All tables use TEXT (UUID) primary keys for distributed systems compatibility
2. *Foreign Keys*: Enforced for referential integrity
3. *Search Indexes*: Cover common query patterns (email lists, search, filtering)
4. *Composite Indexes*: Used where queries filter by multiple columns
5. *Covering Indexes*: Include frequently accessed columns to avoid table lookups

* Data Integrity Rules

** Constraints
1. *Cascade Deletes*: Account deletion cascades to emails, folders, etc.
2. *Soft Deletes*: Emails use ~is_deleted~ flag instead of hard deletion
3. *Unique Constraints*: Prevent duplicate accounts, provider IDs
4. *Check Constraints*: Validate enums and business rules at database level

** Migration Best Practices
- [ ] Always use transactions for schema changes
- [ ] Test migrations on copy of production data
- [ ] Include rollback scripts
- [ ] Version all schema changes
- [ ] Document breaking changes

* Performance Considerations

** Optimization Strategies
1. *Partitioning*: Consider partitioning emails table by account_id for large datasets
2. *Archival*: Move old emails to archive tables after configurable period
3. *Vacuum*: Regular VACUUM operations to maintain performance
4. *Statistics*: Keep SQLite statistics updated with ANALYZE
5. *Connection Pool*: Use connection pooling with appropriate limits

** Query Optimization
- [ ] Use EXPLAIN QUERY PLAN for complex queries
- [ ] Monitor slow queries
- [ ] Add indexes based on actual usage patterns
- [ ] Consider denormalization for read-heavy operations

* Backup Strategy

** Configuration
1. *WAL Mode*: Enable Write-Ahead Logging for better concurrency
2. *Checkpoint*: Regular checkpoints to main database file
3. *Backup*: Periodic backups using SQLite backup API
4. *Export*: JSON export functionality for data portability

* Database Size Estimations

** Storage Calculations
*** Per Email
- Metadata: ~2KB per email
- Search index: ~1KB per email
- Total: ~3KB per email in database

*** Per Account (10,000 emails)
- Database size: ~30MB
- Search index: ~10MB
- File storage: ~1GB (assuming 100KB average email size)

*** Scaling Considerations
- 100K emails: ~300MB database
- 1M emails: ~3GB database
- Consider partitioning at 500K+ emails per account
